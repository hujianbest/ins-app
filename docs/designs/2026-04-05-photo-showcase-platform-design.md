# 摄影作品展与约拍网站实现设计

- **状态：** 已批准
- **主题：** 摄影作品展与约拍平台
- **输入规格：** `docs/specs/2026-04-05-photo-showcase-platform-srs.md`

## 概述

本设计面向一个以视觉展示为核心的摄影作品展与约拍网站。系统需要同时满足三类诉求：

1. 未登录访客可低门槛浏览首页、作品、摄影师主页、模特主页和约拍诉求。
2. 注册摄影师与模特可维护自己的展示主页、上传图片作品或展示内容、发布约拍诉求。
3. 登录用户可点赞作品、收藏主页，并通过站内方式发起联系。

首版重点是“展示 + 联系”的闭环，而非交易闭环。因此设计优先保障以下目标：

- 公共页面的视觉表现和加载体验
- 内容模型对摄影师/模特主页的支撑能力
- 登录后互动能力的闭环
- 后续可平滑扩展为通知、双身份、推荐策略等增强功能

## 设计驱动因素

### 需求驱动

- 首版必须同时支持摄影师与模特两类展示主页。
- 未登录用户必须能浏览公开内容。
- 联系方式限定为站内联系。
- 约拍诉求最小必填信息为 `城市` 与 `时间`。
- 首页和个人主页应具备“全屏视觉冲击”的视觉表达能力。

### 非功能驱动

- 公共页面需要良好的首屏加载与 SEO 友好性。
- 图片内容是核心资产，系统需要稳定支撑图片上传、存储、展示与裁切。
- 交互链路较多，但首版不希望拆成过重的多服务系统。
- 后续需要保留扩展空间，用于通知中心、推荐排序和双身份能力。

### 当前技术上下文假设

- 当前主工作区内尚无明确成型的业务架构约束，因此本设计按绿地项目制定。
- 设计默认采用单仓内 Web 应用优先方案，以降低首版交付复杂度。

## 候选方案

### 方案 A：单体式 Next.js 全栈应用

#### 如何工作

- 使用 Next.js App Router 同时承载公共页面、后台管理页面和服务端接口。
- 使用关系型数据库存储用户、主页、作品、互动和约拍诉求。
- 使用对象存储保存图片文件。
- 通过服务端渲染或静态生成承载首页、公开主页、作品详情和诉求详情。

#### 优点

- 适合内容展示型网站，兼顾 SSR、SEO 和交互页面开发效率。
- 单仓单应用部署简单，首版上线链路最短。
- 服务端与前端同构，角色鉴权、数据读取和表单提交流程一致。
- 便于先做出高质量视觉页面，再逐步补齐后台能力。

#### 缺点

- 业务复杂度上升后，前后端边界需要靠良好模块化约束保持清晰。
- 如果后期出现高并发媒体处理需求，需额外拆分异步处理链路。

#### 适配度

- 与首版“强展示、轻交易、快速闭环”的目标高度匹配。

### 方案 B：前后端分离 SPA + API 服务

#### 如何工作

- 使用 React/Vue 单页前端承载页面。
- 独立 Node.js API 服务负责用户、作品、互动、联系和诉求接口。
- 公共页面通过客户端拉取数据完成渲染。

#### 优点

- 前后端职责边界天然清晰。
- 后续扩展 App、小程序或其他前端时更灵活。

#### 缺点

- 公共内容的 SEO、首屏体验与页面性能优化成本更高。
- 首版需要维护两套工程和部署链路，交付复杂度上升。

#### 适配度

- 可行，但对首版过重。

### 方案 C：CMS 驱动的展示站 + 独立互动服务

#### 如何工作

- 展示主页和首页主要由 CMS 驱动。
- 点赞、收藏、联系与约拍诉求通过外接业务服务实现。

#### 优点

- 静态展示内容搭建快。
- 编辑运营精选内容较方便。

#### 缺点

- 用户生成主页、互动、登录与站内联系并非 CMS 强项。
- 用户内容与业务行为分裂，数据一致性和体验统一性较差。

#### 适配度

- 不适合作为本项目首版主方案。

### 推荐方案

推荐采用 **方案 A：单体式 Next.js 全栈应用**。

原因：

- 最符合公开展示站点的 SEO 和首屏需求。
- 能以较低复杂度同时覆盖公开浏览、用户后台与基础互动能力。
- 更容易把首页和个人主页做成高质量、沉浸式的视觉体验。
- 便于后续逐步抽离媒体处理、消息通知或推荐服务，而不是一开始就为未来复杂度付出过高成本。

## 选定方案

### 总体架构

系统采用以下逻辑分层：

1. **展示层**
   - 公共首页
   - 摄影师主页
   - 模特主页
   - 作品详情页
   - 约拍诉求列表与详情页
   - 登录后个人控制台与收件箱

2. **应用层**
   - 认证与会话服务
   - 主页服务
   - 作品服务
   - 点赞与收藏服务
   - 站内联系服务
   - 约拍诉求服务
   - 首页编排服务
   - 媒体资源服务

3. **数据层**
   - 关系型数据库
   - 图片对象存储
   - 可选缓存层（首版可不强依赖）

### 关键设计决策

- **账号首版采用单主身份模型。**
  用户注册时选择摄影师或模特之一；若后续需要双身份，可在资料模型上扩展为多角色绑定。此决策可显著降低首版账号、权限、主页编辑和诉求发布规则的复杂度。

- **公开内容优先服务端渲染。**
  首页、个人主页、作品详情、诉求详情优先采用 SSR/ISR，以保证首屏视觉与可发现性。

- **互动功能要求登录，但浏览不要求登录。**
  公开浏览最大化降低门槛，点赞、收藏、联系等动作通过统一登录闸口拦截。

- **站内联系采用“会话线程 + 消息”模型。**
  首版只做站内消息，不做外部联系方式暴露，不做实时在线要求。

- **首页精选采用“人工精选位 + 最新内容兜底”编排策略。**
  这样能满足视觉质量控制，也能避免首版就引入复杂推荐算法。

## 架构设计

### 页面与路由结构

- `/`：公共首页
- `/photographers/[slug]`：摄影师主页
- `/models/[slug]`：模特主页
- `/works/[workId]`：作品详情
- `/opportunities`：约拍诉求列表
- `/opportunities/[postId]`：约拍诉求详情
- `/login`、`/register`：认证入口
- `/studio`：登录后控制台首页
- `/studio/profile`：个人主页编辑
- `/studio/works`：作品管理
- `/studio/opportunities`：约拍诉求管理
- `/inbox`：站内联系收件箱

### 视觉布局设计原则

- 首页首屏使用全屏 Hero 区域，承载大图、精选标题和核心导航入口。
- 个人主页以沉浸式封面、图片瀑布/画册式展示和简洁文字信息为主。
- 页面信息分层应避免密集表格感，优先采用视觉模块和内容留白。
- 交互入口应尽量在视觉不被破坏的前提下融入页面，例如悬浮按钮、轻量动作条、局部展开面板。

### 主要运行流程

#### 公开浏览流程

1. 访客访问首页。
2. 首页编排服务聚合精选主页、精选作品和约拍诉求入口。
3. 访客进入个人主页、作品详情或诉求详情。
4. 若访客执行点赞、收藏、联系等动作，则被引导登录。

#### 作品发布流程

1. 摄影师或模特登录并进入控制台。
2. 上传图片并填写展示内容元信息。
3. 媒体资源服务完成文件保存与图片展示地址生成。
4. 作品服务写入作品记录，并将其关联到当前用户主页。
5. 公开主页可读取并展示该内容。

#### 站内联系流程

1. 登录用户在主页或诉求详情页点击联系。
2. 联系服务根据发起方与接收方建立或复用联系线程。
3. 首条消息写入线程。
4. 接收方在 `inbox` 中查看线程与消息。

#### 约拍诉求发布流程

1. 摄影师或模特登录并进入诉求管理页。
2. 填写 `城市`、`时间` 等首版字段并发布。
3. 诉求服务保存记录并公开展示。
4. 浏览者进入诉求详情并发起站内联系。

## 模块职责

### Auth 模块

- 处理注册、登录、登出和会话校验。
- 维护用户基础身份信息与主身份类型。

### Profile 模块

- 管理摄影师/模特主页资料。
- 提供主页公开读取与后台编辑接口。
- 负责主页 slug、封面、头像、简介和展示排序配置。

### Portfolio 模块

- 管理图片作品或展示内容的元数据。
- 提供作品创建、编辑、删除、公开查询与详情查询能力。

### Media 模块

- 接收图片上传。
- 执行图片元数据校验、存储寻址和展示地址生成。
- 支撑封面图、作品图、缩略图等不同场景的资源输出。

### Engagement 模块

- 管理点赞记录与主页收藏记录。
- 提供登录态下的动作接口与统计结果。

### Inquiry 模块

- 管理站内联系线程与消息。
- 支撑从主页、作品页或诉求页发起联系。

### Opportunity 模块

- 管理摄影师/模特发布的约拍诉求。
- 提供诉求列表、详情、发布、编辑与下线能力。

### Home Curation 模块

- 管理首页 Hero 与精选内容位。
- 采用人工精选位优先、最新公开内容兜底的策略生成首页展示数据。

## 数据流与控制流

### 核心实体

- `User`
- `Profile`
- `Work`
- `WorkImage`
- `ProfileFavorite`
- `WorkLike`
- `InquiryThread`
- `InquiryMessage`
- `OpportunityPost`
- `HomeFeatureSlot`

### 关系概要

- 一个 `User` 对应一个主 `Profile`。
- 一个 `Profile` 可拥有多条 `Work`。
- 一个 `Profile` 可拥有多条 `OpportunityPost`。
- `WorkLike` 记录用户对作品的点赞关系。
- `ProfileFavorite` 记录用户对主页的收藏关系。
- `InquiryThread` 连接发起人、接收人以及来源对象。
- `InquiryMessage` 属于某个 `InquiryThread`。

### 来源对象约定

站内联系线程需要记录来源，以便接收方理解上下文。来源对象类型至少包括：

- `profile`
- `work`
- `opportunity`

## 接口与契约

### 页面读取契约

- 首页读取接口需返回 Hero 内容、精选主页列表、精选作品列表、精选诉求列表。
- 主页读取接口需返回用户资料、封面信息、作品列表、主页统计摘要、是否可联系。
- 作品详情读取接口需返回作品主体、所属主页摘要、点赞状态与相关推荐入口。
- 诉求详情读取接口需返回发布者摘要、城市、时间和联系入口。

### 写入契约

- 主页编辑接口只允许主页所有者修改其公开资料。
- 作品上传接口要求登录，且仅允许操作本人主页内容。
- 点赞与收藏接口要求登录，且应保证幂等。
- 发起联系接口要求登录，并要求指定来源对象。
- 诉求发布接口要求登录，并校验 `城市` 与 `时间` 为必填。

### 状态与权限约定

- 公开页面仅展示“公开可见”内容。
- 非本人不可编辑他人主页、作品或诉求。
- 未登录用户无写权限。

## 测试策略

### 单元测试

- 主页、作品、点赞、收藏、联系、诉求等领域服务的规则测试。
- 权限判断和输入校验测试。

### 集成测试

- 注册/登录后创建主页、上传作品、发布诉求的主路径。
- 从公开页面触发登录，再完成点赞、收藏、联系的流程。
- 联系线程创建与复用逻辑。

### 端到端测试

- 未登录访客浏览首页并进入主页、作品和诉求详情。
- 摄影师完成注册、主页编辑、作品上传、诉求发布。
- 模特完成注册、主页编辑、展示内容上传、诉求发布。
- 普通登录用户从公开页面发起点赞、收藏与站内联系。

### 视觉回归

- 首页首屏
- 摄影师主页
- 模特主页
- 作品详情页

## 风险与待定问题

- 首版单主身份模型简化了实现，但若业务很快要求双身份，需要补充角色切换与多主页策略。
- 图片资源体量可能快速增长，需在实现阶段预留对象存储与 CDN 能力。
- 站内联系首版若不做通知提醒，消息触达体验会偏弱，但不阻塞核心闭环。
- 首页精选位若完全依赖人工维护，后续运营成本会增加；可在后续增量中引入规则排序或推荐能力。
