# 摄影社区平台需求规格说明

- 状态: 已批准
- 主题: 摄影社区平台（轻量 MVP）
- 评审结论: 通过（已完成人类确认）
- 关联评审: `docs/reviews/spec-review-photography-community-platform.md`

## 1. 背景与问题陈述

当前站点已经具备首页、公开主页、作品详情、诉求列表与详情、登录注册、工作台、点赞 / 收藏 / 站内联系等基础能力，但其产品主线仍然是“作品展与约拍平台”。

这会带来几个问题：

- 当前产品更偏向展示和撮合，缺少摄影社区应有的“发布作品 -> 被发现 -> 建立关系 -> 持续创作”闭环。
- 首页与公开内容虽然已经能浏览，但还没有独立的社区发现入口，用户难以持续探索作品与创作者。
- 现有互动以点赞、收藏、联系为主，尚未形成“关注 + 评论”这种更适合社区沉淀的关系结构。
- 已有“模特 + 约拍诉求”能力仍然有价值，但不应继续与社区主线并列为产品核心。

本轮需要解决的问题，是把当前站点从“摄影作品展与约拍平台”收敛为“摄影社区平台”的轻量 MVP：以首页、发现、作品详情、创作者主页、发布、关注 / 评论为核心能力，同时保留现有“模特 + 约拍诉求”作为次级模块。

## 2. 目标与成功标准

### 2.1 目标

- 把当前产品主线切换为面向摄影作品与创作者关系的社区平台。
- 在首期只交付轻量 MVP，优先完成社区最小闭环，而不是一次性铺开挑战赛、小组、学习或商业化能力。
- 保留现有“模特 + 约拍诉求”能力，但将其降级为次级模块，不再作为首页与主导航核心。

### 2.2 成功标准

- 未登录访客可以浏览首页、发现页、创作者主页和作品详情页。
- 已登录账号可以关注创作者并对作品发表评论。
- 具有创作者资格的账号可以发布图片作品，并在自己的公开主页中展示这些作品。
- 已保留的模特主页与约拍诉求仍可访问和使用，但不会占据社区首页的核心入口。
- 当前轮次不依赖挑战赛、小组、学习内容、会员或商业授权等扩展能力，也能形成清晰可验证的社区 MVP。

## 3. 用户角色与关键场景

### 3.1 主要用户

- 未登录访客：浏览首页、发现页、创作者主页与作品详情。
- 社区成员：指已登录账号。首期继续沿用现有单主身份模型，注册或首次进入登录态时仍需选择 `摄影师` 或 `模特` 之一作为主身份；登录后可浏览、关注、评论、联系并管理自己的公开内容。
- 摄影师创作者：社区主线中的核心创作者角色，可维护公开主页、发布公开作品，并作为首页与发现页优先承接的内容供给角色。
- 模特创作者：次级创作者角色，继续保留公开主页与展示内容发布能力，但不作为首页与主导航的核心叙事对象。
- 平台编辑 / 运营：维护首页精选与发现入口中的人工精选内容。

### 3.2 次级用户

- 模特：继续保留个人主页与公开展示能力，但不作为本轮社区主线的核心角色。
- 合作发布者：继续保留约拍诉求发布与公开浏览能力，但作为次级模块存在。

### 3.3 关键场景

- 访客打开首页后，可以明确感知这是一个摄影社区，并继续进入发现页、作品详情或创作者主页。
- 注册成员浏览作品后，可以关注喜欢的创作者，并在作品详情页留下评论。
- 创作者可以发布新的作品内容，并在公开主页中持续积累代表作。
- 对合作仍有需求的用户，可以继续访问模特主页和约拍诉求，但不会被这些能力主导整个产品结构。

## 4. 范围

- 首页社区化，突出作品与创作者发现入口
- 独立发现页
- 创作者公开主页
- 作品发布、编辑与公开展示
- 作品详情页
- 关注创作者
- 作品评论
- 首页 / 发现页人工精选内容维护能力
- 保留模特主页能力
- 保留约拍诉求列表、详情与站内联系能力
- 保留现有点赞、收藏、站内联系等兼容能力，但不把它们定义为本轮主目标

## 5. 范围外内容

- 挑战赛 / 征集系统
- 小组 / Spaces
- 学习内容、访谈、教程体系
- 系列作品系统
- EXIF / 器材 / 地区等高级发现维度
- 高级创作者中心、成长任务、数据分析
- 会员订阅、商业授权、品牌合作、衍生品售卖
- 通知中心
- 独立运营后台或人工精选专用管理界面
- 评论编辑 / 删除 / 举报 UI
- 内容审核后台
- 在线支付
- 移动 App

## 6. 术语与定义

- 摄影社区平台：以作品发布、内容发现、创作者关系和社区互动为核心的摄影产品形态。
- 主身份：首期账号必须选择的唯一身份类型，当前只包含 `摄影师` 或 `模特`。
- 社区成员：任意已登录账号。首期所有社区成员都基于一个主身份存在。
- 创作者：拥有 `摄影师` 或 `模特` 主身份、具备维护公开主页和发布公开展示内容资格的账号。
- 创作者主页：承载创作者公开身份、简介与作品列表的公开页面。
- 发现页：用于浏览精选、最新与已关注创作者内容的公共入口页。
- 人工精选：由平台编辑 / 运营维护的精选内容输入。首期允许通过静态配置或受限内部维护方式完成，不要求独立运营后台。
- 次级模块：本轮继续保留，但不进入首页核心叙事和主导航中心的能力模块。

## 7. 功能需求

### FR-001 账号与访问控制

#### 需求说明

系统应继续支持公开浏览与登录后互动的双层访问模式。首期仍沿用现有单主身份模型，账号在注册或首次进入登录态时需要选择 `摄影师` 或 `模特` 之一作为主身份；未登录访客可以浏览核心社区内容，而需要建立关系或发布内容的操作需要登录。

#### 验收标准

- Given 未登录访客访问首页、发现页、创作者主页或作品详情，When 页面渲染完成，Then 系统应允许其直接浏览公开内容。
- Given 用户注册或首次进入登录态，When 完成身份选择，Then 系统应要求其选择且仅选择一个主身份：`摄影师` 或 `模特`。
- Given 未登录访客尝试执行关注、评论或发布操作，When 动作触发，Then 系统应引导其先登录。
- Given 已登录账号的主身份为 `摄影师` 或 `模特`，When 进入发布或管理路径，Then 系统应允许其维护公开主页和公开展示内容。

### FR-002 首页社区化

#### 需求说明

系统应将首页定义为摄影社区首页，优先承接品牌表达、精选作品、精选创作者和发现入口，而不是继续以约拍或合作诉求作为首页核心结构。

#### 验收标准

- Given 访客访问首页，When 页面完成渲染，Then 首页应优先展示与摄影作品、创作者或发现相关的核心入口。
- Given 首页保留次级模块入口，When 用户浏览首页，Then 模特主页与约拍诉求不应占据首页首屏主视觉或主导航核心位置。
- Given 用户从首页继续浏览，When 点击主要入口，Then 应可以进入发现页、作品详情、创作者主页或其他社区核心内容。

### FR-003 发现页与公共浏览

#### 需求说明

系统应提供独立发现页，支持用户持续浏览公开作品与创作者内容，而不是仅依赖首页一次性跳转。

#### 验收标准

- Given 访客访问发现页，When 页面渲染完成，Then 系统应展示可持续浏览的公开作品或创作者内容列表。
- Given 已登录社区成员访问发现页，When 需要查看已关注创作者的新内容，Then 系统应提供可识别的“关注中”内容入口或等价浏览方式。
- Given 任意用户浏览发现页，When 需要在不同内容来源间切换，Then 系统至少应支持“精选”“最新”两类公共浏览方式。

### FR-004 创作者主页

#### 需求说明

系统应提供以创作者为中心的公开主页，用于展示创作者身份、简介与公开作品，并允许其他用户建立关注关系。

#### 验收标准

- Given 访客访问创作者主页，When 页面渲染完成，Then 页面应展示创作者名称、简介和公开作品列表。
- Given 创作者已发布公开作品，When 其他用户访问其主页，Then 用户应能从主页进入对应作品详情。
- Given 已登录社区成员访问创作者主页，When 触发关注动作，Then 系统应允许其建立对该创作者的关注关系。

### FR-005 作品发布与作品详情

#### 需求说明

系统应允许创作者发布图片作品，并让这些作品在创作者主页与独立详情页中公开展示。首期作品至少应支持 `草稿` 与 `已发布` 两种状态；只有 `已发布` 作品会进入公开主页、作品详情与发现页。

#### 验收标准

- Given 创作者进入发布或管理路径，When 填写作品基础信息并保存为草稿，Then 系统应记录该作品但不向公开主页、发现页或作品详情开放。
- Given 创作者进入发布或管理路径，When 填写作品基础信息并完成发布，Then 系统应生成一条可公开浏览的作品内容。
- Given 作品已发布，When 其他用户访问创作者主页或发现页，Then 应可以进入对应作品详情页。
- Given 创作者编辑一条已发布作品，When 保存修改，Then 该作品仍应保持公开可见，除非创作者主动将其切回草稿。
- Given 用户访问作品详情页，When 页面渲染完成，Then 页面应展示作品主体、作者信息与评论入口。

### FR-006 关注与评论

#### 需求说明

系统应提供最小可用的社区关系能力，使用户可以关注创作者，并围绕作品进行评论交流。评论首期仅支持纯文本发布，不提供编辑、删除或举报 UI；系统至少需要约束非空输入、基础长度限制和最小失败反馈。

#### 验收标准

- Given 已登录社区成员访问创作者主页，When 触发关注操作，Then 系统应记录该成员对创作者的关注关系。
- Given 已登录社区成员已关注某位创作者，When 再次触发关注入口，Then 系统应允许其取消关注。
- Given 已登录社区成员访问作品详情页，When 提交 1 到 500 个字符之间的纯文本评论，Then 系统应在该作品下展示评论结果。
- Given 已登录社区成员提交空评论或超出长度限制的评论，When 系统校验失败，Then 系统应拒绝提交并给出明确的用户可见错误提示。
- Given 某个作品存在多条评论，When 用户访问作品详情页，Then 系统应按最新优先的顺序展示评论列表。
- Given 未登录访客访问作品详情页或创作者主页，When 触发关注或评论操作，Then 系统应要求其先登录。

### FR-007 人工精选维护

#### 需求说明

系统应支持首页与发现页消费由平台编辑 / 运营维护的人工精选内容，以保证首期社区分发不依赖个性化算法。首期不要求独立运营后台，允许通过静态配置或受限内部维护方式完成精选维护。

#### 验收标准

- Given 平台已维护首页或发现页的人工精选输入，When 页面渲染完成，Then 系统应优先展示这些精选内容。
- Given 某个精选项失效、数量不足或无法命中公开内容，When 页面渲染完成，Then 系统应回退到最新公开内容或输出较少条目，而不应导致页面报错。
- Given 当前轮次仍是轻量 MVP，When 维护人工精选内容，Then 系统不要求提供独立的运营后台界面。

### FR-008 次级合作模块保留

#### 需求说明

系统应继续保留现有模特主页与约拍诉求能力，以保证合作场景不被直接移除，但这些能力在本轮仅作为次级模块存在。

#### 验收标准

- Given 用户通过直接入口或次级入口访问模特主页，When 页面渲染完成，Then 系统应继续支持公开浏览该主页。
- Given 用户访问约拍诉求列表或详情页，When 页面渲染完成，Then 系统应继续支持公开浏览与站内联系。
- Given 用户浏览首页或主导航，When 对比社区核心模块与合作模块，Then 模特主页与约拍诉求不应被表达为本轮产品主线。

## 8. 非功能需求

### NFR-001 图片优先体验

首页、发现页、创作者主页与作品详情页在首屏或首个主要内容区中，必须直接展示 Hero 视觉、作品卡片列表、封面图或作品主体之一；公共浏览页不应要求用户先穿过表单或纯文本说明，才能看到主要视觉内容。

### NFR-002 公开浏览低门槛

社区核心公共内容应维持未登录可浏览，避免用户在建立兴趣之前被登录门槛阻断。

### NFR-003 MVP 范围收敛

本轮规格必须严格收敛到轻量 MVP，不应在当前批准范围中混入挑战赛、小组、学习、会员、商业化或高级发现维度。

### NFR-004 路由稳定性

在切换产品主线时，现有公开浏览路径 `/`、`/photographers/[slug]`、`/models/[slug]`、`/works/[workId]`、`/opportunities`、`/opportunities/[postId]` 必须继续有效；本轮只允许新增社区相关路径，不要求废弃这些既有路径。

### NFR-005 桌面端与移动端基本可用

首页、发现页、创作者主页、作品详情页和基础发布入口应同时支持常见桌面端与移动端浏览；主要内容区与核心操作入口在两类端上都不应出现横向溢出、不可点击或首屏完全不可见的情况。

## 9. 外部接口与依赖

- 依赖现有或后续补齐的登录态与会话能力，支撑关注、评论与发布操作
- 依赖图片上传与公开展示能力，支撑作品发布和详情展示
- 依赖首页 / 发现页中的人工精选输入，以支撑首期非算法化社区分发
- 依赖现有模特主页、约拍诉求与站内联系路径继续工作，作为次级模块保留

## 10. 约束

- 当前轮次必须以摄影社区核心闭环为主，不允许把“模特 + 约拍诉求”重新抬升为并列主线
- 当前轮次继续沿用现有单主身份模型，注册账号在 `摄影师` 与 `模特` 之间二选一
- 当前轮次不以强算法推荐为前提
- 当前轮次不引入挑战赛、小组、学习、会员或商业化系统
- 当前轮次不要求重做全部信息架构与现有公开路由
- 当前轮次不包含交易闭环

## 11. 假设

- 假设现有公开主页、作品详情、认证与登录后管理能力可作为社区版 MVP 的可复用基础
- 假设当前站点的主要内容供给可先由创作者自发发布与人工精选共同支撑
- 假设模特主页与约拍诉求在当前阶段仍有业务价值，但它们的产品优先级低于社区核心模块

## 12. 开放问题

### 阻塞性开放问题

无

### 非阻塞开放问题

- 后续增量中，挑战赛 / 小组 / 学习三类社区增强能力的优先级顺序如何安排
- 现有点赞、收藏与站内联系能力，后续是长期保留为次级互动，还是继续弱化
- 模特主页与约拍诉求在后续版本中，是否会被进一步整合为统一的“合作”次级模块
