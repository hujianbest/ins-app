# 综合摄影平台重发布需求规格说明

- 状态: 已确认
- 主题: Hybrid Platform Relaunch
- 输入来源: `Hybrid Relaunch Plan`

## 1. 背景与问题陈述

当前站点已经具备首页、发现页、创作者主页、作品详情、工作台、关注 / 评论、次级合作入口等基础能力，但整体仍明显停留在轻量 MVP 状态：

- 运行时仍以 demo 级会话与本地 SQLite 为主，云端长期部署与生产运维边界不够清晰。
- 全站缺少成熟的导航壳层、统一的设计令牌和稳定的信息架构，观感更像验证原型而不是正式产品。
- 作品发布、发现、合作与用户关系虽然已经具备雏形，但还没有被收束为“可上线、可持续扩展”的综合平台。
- 初始内容与种子数据仍偏工程样本，不足以支撑首发阶段的视觉质感和引导浏览。

本轮需要把当前站点从“摄影社区轻量 MVP”重定为“可云端部署的综合摄影平台”，采用分阶段上线策略：

- 第一阶段交付成熟的暗色杂志风体验、Docker 部署、基础账号体系、作品与创作者主线、基础搜索、合作线索入口和安全的初始内容。
- 第二阶段再扩展运营后台、消息中心、支付 / 订单和会员等更重的商业能力。

## 2. 目标与成功标准

### 2.1 总体目标

- 将当前 `web` 应用升级为面向真实用户上线的综合摄影平台，而非继续停留在演示站或内部原型状态。
- 保留当前 Next.js 单体的可复用资产，但替换 demo 级运行时与过于简陋的界面表达。
- 通过分阶段范围控制，在保证首发质量的前提下优先完成“可上线、可浏览、可发布、可互动、可留下合作线索”的核心闭环。

### 2.2 阶段策略

- 阶段 1（当前交付目标）：
  - 成熟的暗色杂志风界面与统一壳层
  - Docker 云部署基线
  - 真实账号 / 会话、作品发布、发现流、创作者主页、关注 / 评论
  - 基础搜索
  - 合作线索入口
  - 安全的初始引流内容
- 阶段 2（后续 backlog）：
  - 完整运营后台
  - 线程式消息中心
  - 支付 / 订单 / 会员
  - 更完整的商业合作流程与运营能力

### 2.3 阶段 1 成功标准

- 站点可通过 Docker 在云服务器上稳定启动，并具备明确的环境变量、数据路径和健康检查约定。
- 未登录访客可以浏览首页、发现页、搜索结果、创作者主页、作品详情和合作入口。
- 已登录成员可以注册 / 登录、维护个人资料、发布作品、关注创作者并发表评论。
- 首页、发现页、创作者主页和作品详情页在视觉层级、导航结构和内容密度上表现为成熟产品，而非 MVP 样机。
- 站点自带一批安全来源的高质量初始内容，首屏与核心浏览路径不再依赖工程测试样本。
- 当前阶段不依赖运营后台、消息中心或支付能力，也能独立上线并形成完整主线。

## 3. 用户角色与关键场景

### 3.1 主要用户

- 未登录访客：浏览首页、发现页、搜索结果、创作者主页、作品详情与合作入口。
- 平台成员：注册并登录后，可关注、评论、收藏、管理资料与公开内容。
- 创作者：具有 `摄影师` 或 `模特` 主身份的成员，可维护主页、发布作品、展示代表作并接受合作线索。
- 客户 / 品牌 / 潜在合作方：希望浏览作品与创作者资料，并通过合作入口发起联系。
- 平台运营：阶段 1 不要求完整后台，但仍需要通过有限配置入口维护首页 / 发现页精选与初始内容。

### 3.2 关键场景

- 访客打开首页后，应立刻感知这是一个成熟的摄影平台，可继续进入发现、搜索、创作者和作品浏览路径。
- 成员注册或登录后，可以进入工作台维护个人资料、发布作品，并在公开面上持续积累展示内容。
- 访客或已登录成员浏览某个作品或创作者后，可以继续搜索相关内容、关注创作者，或留下合作意向。
- 首发阶段即使尚未建设完整后台和支付流程，站点也应具备真实部署、真实浏览和真实发布的基础能力。

## 4. 阶段 1 范围

- 全站统一导航、页脚、用户入口与暗色杂志风设计系统
- 首页重构
- 发现页重构
- 基础搜索页与搜索入口
- 创作者公开主页
- 作品详情页
- 工作台资料维护
- 作品发布、编辑、草稿 / 发布状态
- 关注 / 评论 / 收藏等基础互动
- 合作线索入口与联系动作
- 账号注册、登录、登出与安全会话
- Docker 部署、环境变量契约、健康检查
- 安全的初始引流内容与精选配置

## 5. 阶段 1 范围外内容

- 完整运营后台与审核工作台
- 线程式站内消息中心
- 支付、订单、会员与结算
- 高级搜索排序、推荐算法、独立检索服务
- 原生移动 App
- 挑战赛、小组、学习内容体系
- 评论举报、内容审核流与复杂风控后台

## 6. 术语与定义

- 综合摄影平台：同时承接作品浏览、创作者展示、社交互动与合作线索的产品形态。
- 阶段 1：当前首发目标，以可上线、可部署、可浏览、可发布为第一优先级。
- 合作线索：用户围绕某个创作者或作品发起的联系意向，阶段 1 只要求形成稳定入口与落库链路。
- 初始引流内容：为首发阶段准备的授权图片、虚构作者资料、作品文案与精选配置，用于提升首屏质量与浏览体验。

## 7. 功能需求

### FR-001 账号与安全会话

#### 需求说明

系统应提供真实可用的注册、登录、登出与会话能力，替换当前 demo 级角色 cookie 模式。阶段 1 保留 `摄影师` 与 `模特` 两种主身份，但不再允许仅靠写入角色字符串构造登录态。

#### 验收标准

- Given 访客访问注册页，When 提交合法邮箱、密码与主身份，Then 系统应创建账号并建立安全会话。
- Given 已存在账号的用户访问登录页，When 提交正确凭据，Then 系统应建立安全会话并进入工作台。
- Given 用户登出，When 动作完成，Then 受保护的工作台页面应重新要求登录。
- Given 非法 cookie 或无效会话访问受保护页面，When 页面渲染，Then 系统应将其视为未登录。

### FR-002 Docker 部署与运行时基线

#### 需求说明

系统应可以通过 Docker 在云服务器上部署运行，并明确环境变量、持久化数据、健康检查与启动约定。

#### 验收标准

- Given 运维使用项目提供的容器化配置构建镜像，When 执行生产启动命令，Then Next.js 应能够在容器内成功启动。
- Given 生产环境缺少关键变量，When 应用启动，Then 系统应尽早失败并输出明确错误，而不是静默回退到不安全默认值。
- Given 运维检查健康接口，When 应用和数据源可用，Then 系统应返回可判读的健康状态。
- Given 生产环境发生重启，When 站点恢复，Then 已持久化的内容与账号状态不应因容器重建而丢失。

### FR-003 全站壳层与视觉系统

#### 需求说明

系统应建立统一的暗色杂志风设计系统、全站导航壳层和稳定的页面层级，让首页、发现页、作品详情、创作者主页和工作台具备一致的高级产品感。

#### 验收标准

- Given 用户访问任意核心页面，When 页面加载完成，Then 页面应展示统一的导航、品牌、主要入口和页脚结构。
- Given 用户在首页、发现页、创作者主页和作品详情之间切换，When 浏览进行中，Then 视觉语言、按钮样式、卡片结构与排版层级应保持一致。
- Given 页面存在图片内容，When 首屏渲染，Then 图片、标题和主要行为入口应优先呈现，而不是先出现低密度表单或工程占位文本。

### FR-004 首页与发现页

#### 需求说明

系统应将首页定义为品牌与高质量内容的入口，并通过发现页与搜索入口形成可持续浏览路径。首页需要同时承接精选内容、最新内容、创作者入口和合作入口，但必须有明确主次关系。

#### 验收标准

- Given 访客访问首页，When 页面渲染完成，Then 首页应优先展示高质量作品与创作者内容，并提供清晰的发现 / 搜索 / 发布入口。
- Given 用户访问发现页，When 页面渲染完成，Then 系统应展示可持续浏览的精选内容、最新发布和可识别的内容分区。
- Given 用户使用搜索入口，When 输入关键词，Then 系统应能返回作品、创作者或合作内容的匹配结果。
- Given 次级合作内容存在，When 首页渲染，Then 合作内容应作为次级入口存在，而不应取代作品与创作者成为首屏主叙事。

### FR-005 创作者主页与作品详情

#### 需求说明

系统应提供成熟的创作者主页和作品详情页，承接创作者展示、作品浏览、关注关系、合作入口与相关内容延展。

#### 验收标准

- Given 访客访问创作者主页，When 页面渲染完成，Then 应能看到创作者头像 / 头图区域、简介、代表作品、统计或摘要信息以及合作入口。
- Given 访客访问作品详情页，When 页面渲染完成，Then 应能看到作品主体、作者摘要、互动区和进一步浏览入口。
- Given 创作者存在已发布作品，When 其他用户访问其主页，Then 用户应可从主页进入对应作品详情。
- Given 某个作品为草稿，When 访客尝试通过公开路由访问，Then 系统应隐藏该作品或返回不可访问结果。

### FR-006 作品发布与资料维护

#### 需求说明

系统应允许创作者在工作台中维护公开资料、创建作品草稿、发布作品并继续编辑已发布内容。阶段 1 至少要稳定支持封面图引用、作品标题、分类、简介、详细说明与发布状态。

#### 验收标准

- Given 已登录创作者访问工作台资料页，When 提交合法资料，Then 系统应保存并在公开主页中反映最新内容。
- Given 已登录创作者访问作品管理页，When 保存草稿，Then 作品应仅在工作台可见。
- Given 已登录创作者发布作品，When 发布完成，Then 作品应进入公开主页、发现流和搜索结果。
- Given 创作者编辑已发布作品，When 保存修改，Then 作品默认仍保持公开可见，除非用户主动回退到草稿。

### FR-007 社交互动

#### 需求说明

系统应保留并强化基础社交互动，至少包括关注创作者、评论作品与收藏作品。阶段 1 不要求评论治理后台或复杂通知系统，但必须保证最小可用与清晰反馈。

#### 验收标准

- Given 已登录成员访问创作者主页，When 触发关注或取消关注，Then 系统应记录对应关系。
- Given 已登录成员访问作品详情页，When 提交合法评论，Then 系统应按最新优先展示评论。
- Given 未登录访客尝试关注、评论或收藏，When 动作触发，Then 系统应要求其先登录。
- Given 评论为空或超出长度限制，When 提交失败，Then 系统应给出明确用户可见错误。

### FR-008 基础搜索

#### 需求说明

系统应提供基础搜索，帮助用户在作品、创作者和合作内容之间快速定位目标。阶段 1 以关键词搜索和有限过滤为主，不要求高级排序引擎。

#### 验收标准

- Given 用户在搜索框输入关键词，When 提交搜索，Then 系统应返回包含作品与创作者的匹配结果。
- Given 合作内容命中关键词，When 搜索结果渲染，Then 系统应能在结果中识别并展示合作相关条目。
- Given 当前没有命中结果，When 页面渲染，Then 系统应展示稳定空态而不是报错。

### FR-009 合作线索入口

#### 需求说明

系统应保留并升级合作相关入口，让用户可以围绕创作者或作品发起联系。阶段 1 只要求形成高质量线索收集与联系链路，不要求支付或复杂订单状态机。

#### 验收标准

- Given 访客浏览创作者主页或作品详情，When 点击合作入口，Then 系统应进入明确的联系动作或表单入口。
- Given 用户提交合法合作信息，When 动作成功，Then 系统应持久化该条线索并给出成功反馈。
- Given 当前阶段未启用支付功能，When 用户浏览合作链路，Then 系统不应展示伪造的下单或支付完成状态。

### FR-010 初始引流内容与精选维护

#### 需求说明

系统应提供安全来源的高质量初始内容，以支撑首发阶段的视觉质量和浏览密度；同时保留有限的精选维护能力，用于首页与发现页的首屏编排。

#### 验收标准

- Given 系统初始化阶段执行内容导入，When 种子导入完成，Then 站点应拥有一批高质量作品、创作者资料和精选配置。
- Given 初始内容来自外部素材，When 导入到站点，Then 必须只使用授权明确的图片资源与虚构资料，不得抓取未知版权内容。
- Given 精选配置命中失效目标，When 页面渲染，Then 系统应回退到可公开浏览的内容，而不是报错或空白。

## 8. 非功能需求

### NFR-001 视觉成熟度

首页、发现页、创作者主页、作品详情和工作台的排版、留白、图片权重、交互按钮与层级关系必须达到成熟产品水平，避免出现明显的演示页、工程页或临时卡片堆叠观感。

### NFR-002 图片优先与响应式可用性

核心公共页面必须在桌面端和移动端都优先呈现图片和主要信息区，不允许出现首屏完全被文本占满、横向溢出或核心按钮不可点击的情况。

### NFR-003 云部署可操作性

项目必须具备明确的 Docker 构建和启动路径，并通过环境变量契约、健康检查和持久化策略支撑单实例云端部署。

### NFR-004 会话与数据安全

阶段 1 虽不要求复杂安全体系，但至少应使用安全 cookie、服务端校验、持久化账号与会话数据，并避免继续把角色字符串当作唯一登录依据。

### NFR-005 数据与资源可迁移性

作品、创作者、合作线索、关注、评论与账号数据应通过 repository 边界访问，后续可以迁移到托管数据库或对象存储，而不需要重写页面层。

### NFR-006 初始内容合法性

首发阶段用于引流的图片和资料必须具有明确的安全来源，禁止使用来路不明的抓取内容作为生产种子。

## 9. 外部接口与依赖

- Docker 运行环境与云服务器
- 持久化数据存储（首发可基于单实例持久化方案，后续可迁移到托管 SQL）
- 图片资源存储与公开访问能力
- 安全的环境变量管理
- 授权图片来源与种子内容生成流程

## 10. 约束

- 当前代码基座继续采用 `web` 下的 Next.js 单体应用。
- 当前阶段优先上线质量和产品完成度，不把完整后台、消息和支付挤进首发版本。
- 当前阶段继续沿用 `摄影师` / `模特` 双主身份模型。
- 当前阶段允许使用有限配置维护首页 / 发现页精选，不强制先做完整后台。
- 当前阶段的合作链路只要求线索入口和联系闭环，不扩展到支付与履约。

## 11. 假设

- 现有公开路由、社区读写边界和工作台页面可以作为重发布的代码基础继续演进。
- 阶段 1 的首发流量与业务复杂度仍可由单体部署架构承接。
- 初始引流内容可以通过授权图片与虚构资料组合生成，不依赖真实用户内容也能形成较好的首屏质量。

## 12. 开放问题

### 阻塞性开放问题

无

### 非阻塞开放问题

- 生产环境最终优先接入哪家托管 SQL 与对象存储服务
- 阶段 1 的合作线索是否需要同步邮件提醒，还是先只保留站内落库
- 阶段 2 的运营后台优先覆盖精选维护、审核还是账号管理
