## 1. 概述

本计划面向第二轮“首页与发现体验增强”增量，目标是在不引入个性化推荐和独立运营后台的前提下，完成首页发现分区、编排规则和首页 UI 集成。

本轮任务拆解遵循以下原则：

- 优先把首页发现能力拆成“类型/配置 -> 编排逻辑 -> 页面集成 -> 回归验证”四段
- 每个任务都应形成单任务可推进、可验证、可提交的闭环
- 先固化规则层，再接入页面层，避免在 UI 实现时一边补规则一边补结构

## 5. 任务拆解

### T13. 建立首页发现类型与精选配置契约

- 目标: 建立 `home-discovery` 模块的类型系统、精选配置结构和样本数据中的时间键/定位支撑。
- 依赖: 已批准规格与设计
- 触碰工件: `web/src/features/home-discovery/types.ts`、`web/src/features/home-discovery/config.ts`、`web/src/features/showcase/types.ts`、`web/src/features/showcase/sample-data.ts`
- 测试设计种子:
  - 主要行为: 精选配置可表达 `works / profiles / opportunities` 三类分区，且 `profiles` 使用 `role + slug` 联合定位。
  - 关键边界或负向场景: 精选配置命中不存在对象时，后续 resolver 能跳过无效精选位。
  - fail-first 优先点: 先为类型/配置消费路径补一个最小契约测试，证明 `profiles` 分区不能只靠单一 `slug`。
- 验证方式: 类型检查、首页发现相关单元测试或最小契约测试通过。
- 预期证据: `npm run test` 中新增首页发现契约相关测试通过；必要时 `npm run build` 类型检查通过。
- 完成条件: 首页发现类型、配置结构和样本数据时间键支撑已经存在，后续 resolver 无需再回头补输入契约。

### T14. 建立首页发现卡片适配层

- 目标: 将作品、主页、诉求三类公开内容统一适配为首页发现卡片视图模型。
- 依赖: `T13`
- 触碰工件: `web/src/features/home-discovery/adapters.ts`、相关测试文件、必要时 `web/src/features/showcase/sample-data.ts`
- 测试设计种子:
  - 主要行为: 三类公开内容都能被映射为带 `href`、标题、摘要、badge 的首页卡片。
  - 关键边界或负向场景: 摄影师主页与模特主页的链接路径应区分正确；诉求卡片应指向详情页而不是列表页。
  - fail-first 优先点: 先对每类内容写一个 adapter 输出断言，再补最小映射实现。
- 验证方式: adapter 单元测试通过。
- 预期证据: `npm run test` 中首页卡片适配测试通过。
- 完成条件: 首页发现层不再直接依赖原始公开内容结构渲染卡片。

### T15. 实现首页发现 resolver 规则

- 目标: 实现“人工精选优先 + 最新公开内容兜底 + 同分区去重 + 不足 3 条允许少量输出”的编排逻辑。
- 依赖: `T13` `T14`
- 触碰工件: `web/src/features/home-discovery/resolver.ts`、相关测试文件、必要时 `web/src/features/showcase/sample-data.ts`
- 测试设计种子:
  - 主要行为: resolver 为三个分区各输出最多 3 条结构化卡片。
  - 关键边界或负向场景: 精选不足时补足；精选与兜底去重；某类内容全量为空时返回空态 section 而不是抛错。
  - fail-first 优先点: 先用精选不足/重复内容/空分区三组测试锁定规则，再补 resolver。
- 验证方式: resolver 单元测试通过。
- 预期证据: `npm run test` 中首页发现 resolver 规则测试通过。
- 完成条件: 首页发现编排规则已被集中实现，页面层无需自行补精选/兜底逻辑。

### T16. 集成首页发现分区到首页页面

- 目标: 在不破坏 Hero 首屏主视觉的前提下，将三个发现分区接入首页，并渲染空态、卡片与公开跳转入口。
- 依赖: `T15`
- 触碰工件: `web/src/features/home-discovery/home-discovery-section.tsx`、`web/src/app/page.tsx`、首页相关测试文件
- 测试设计种子:
  - 主要行为: 首页渲染 Hero + 三个发现分区，且每个分区内容可点击进入对应公开详情。
  - 关键边界或负向场景: 某类内容全量为空时，分区壳与空态提示仍应存在；未登录访客访问不受阻断。
  - fail-first 优先点: 先扩展首页渲染测试，确认分区标题、链接与空态提示，再补页面集成。
- 验证方式: 首页渲染测试通过。
- 预期证据: `npm run test` 中首页渲染测试通过，且 `npm run build` 仍成功。
- 完成条件: 首页发现增强已对用户可见，Hero 位置和现有公开路由未被破坏。

### T17. 完成首页发现增强回归收口

- 目标: 用新增测试与全量验证确认首页发现增强没有破坏现有首页体验、公开导航和构建健康。
- 依赖: `T16`
- 触碰工件: 首页相关测试文件、必要时回归记录与状态工件
- 测试设计种子:
  - 主要行为: 首页发现规则测试、首页渲染测试与现有公开浏览入口共同通过。
  - 关键边界或负向场景: 空态分区、重复内容去重、Hero 首屏存在、未登录访客访问公开内容不受阻断。
  - fail-first 优先点: 若前序任务未完全覆盖视觉/导航回归，可补一条明确回归测试后再修实现。
- 验证方式: `npm run test && npm run lint && npm run build`
- 预期证据: 当前最新代码状态下，首页发现相关测试和全站静态检查/构建全部通过。
- 完成条件: 本轮实现具备进入质量链与完成门禁的基础验证证据。

## 6. 依赖与关键路径

- `T13 -> T14 -> T15 -> T16 -> T17`
- 关键路径为“输入契约 -> adapter -> resolver -> 首页集成 -> 全量验证”
- `T15` 是本轮风险最高任务，因为它同时承载精选优先、最新兜底、去重和空态契约

## 7. 完成定义与验证策略

- 每个任务都必须具备可运行的验证方式，而不是只给视觉描述。
- 首页发现规则优先用单元测试锁定，再接页面渲染测试。
- 页面集成后必须执行至少一次 `npm run build`，确认首页增强没有破坏 Next.js 构建。
- 进入实现阶段后仍遵守 fail-first；本计划中的测试设计种子只为 `ahe-test-driven-dev` 提供起点，不替代真人测试设计确认。

## 8. 当前活跃任务选择规则

## 9. 风险与顺序说明

- 当前首页尚无首页发现模块，若直接从页面层开始改，会导致页面 JSX 与规则编排耦合，因此必须先做 `T13` ~ `T15`。
- “最新内容”依赖稳定时间键；若样本数据时间字段不统一，`T13` 必须先补齐统一支撑。
- 空态分区规则会直接影响 `FR-001` 的验收口径，因此 `T15` 和 `T16` 之间不能跳过空态验证。
